# AGENTS.md

本文件是当前知识库仓库的唯一稳定 Agent 主规则入口。Codex、Claude Code、Hermes/OpenClaw 等 AI Agent 进入本仓库时，应优先遵循本文件。

本仓库是影刀 xbot / AI 开发知识库，不是真实影刀项目根目录。实际开发影刀项目时，Agent 应打开真实影刀项目根目录工作。

`templates/` 只保存可选的通用 Agent 规则模板，不参与本知识库检索和规则加载。

## 使用顺序

1. 先看 `llms.txt`，了解项目总览、目录结构和文档入口。
2. 进入 `xbot-api-docs/` 后，再看该目录下的 `AGENTS.md` 和 `llms.txt`。
3. 任务涉及影刀 API、xbot 指令、开发流程或文档维护时，Agent 应主动读取对应文档；`llms.txt` 不是所有 Agent 都会自动读取。

## 使用原则

1. 写影刀编码版代码前，优先查 `xbot-api-docs`。
2. 不确定 API 行为时，不要猜，标注“需运行验证”。
3. 市场指令参数不明确时，先按 `xbot-api-docs/docs/debug/market-extension-source.md` 的方法排查源码。
4. 价格采集优先使用 XPath、CSS、页面接口或明确规则取数，不要让 Agent 自由猜页面价格。
5. 涉及浏览器、URL、网页业务功能时，除非用户明确要求，不要使用 `requests`、`httpx`、`aiohttp`、`urllib` 或其他网络相关 Python 库；只使用 `xbot.web` 及其浏览器对象能力处理页面、请求、Cookie、网络监听和下载。
6. 代码优先保证可读性，变量名、流程和判断逻辑要直观。
7. 不要过度工程化，不要为了简单任务引入复杂封装、抽象层、配置系统或通用框架。
8. 只基于当前业务、当前代码、当前页面和当前需求修改。
9. 不要优化未知问题，不要提前处理未出现、未验证、低概率的场景。
10. 不要做无关重构，不要改动与当前任务无关的代码、格式、命名或结构。
11. 修改代码时优先做最小可行改动，确保每一处改动都能对应当前需求。
12. 如果发现潜在问题，可以指出，但不要擅自扩展修改范围。
13. 页面字段、价格口径、按钮含义必须以当前业务规则为准，不要自行推断。
14. 可视化界面能跑不代表编码版参数一致，编码版参数必须以文档或源码为准。
15. 能用元素操作就不要优先坐标点击。
16. 中文或长文本输入优先考虑剪贴板输入。
17. 表格大量写入时，优先考虑二维数组批量写入。
18. 不要提交影刀运行包源码、反编译源码、敏感路径、账号信息或 token。
19. 不要编造不存在的功能、命令、目录、文件或 API。
20. 钉钉 AI 表格记录默认按 `record["fields"]` 取值；读取记录列表时先走 `data` 层，不要直接对顶层结果 `.get("records")`。
21. 遇到多选字段时，先按 `{"name": "...", "id": "..."}` 结构理解显示值，但真正开发前必须实测字段真实结构，不要凭经验假设字段格式。

## 代码风格要求

1. 代码要简单、直接、容易阅读。
2. 优先顺序式流程，不要为了少量逻辑拆很多函数。
3. 只有当重复明显、逻辑复杂或当前任务明确需要时，才抽函数。
4. 不要默认添加复杂异常体系、重试框架、日志框架或配置层。
5. 异常处理只覆盖当前业务中明确需要处理的失败情况。
6. 注释只解释业务关键点、特殊页面逻辑或不直观的 API 行为，不要写无意义注释。
7. 保持现有项目风格，不要无关格式化整个文件。
8. 涉及“业务处理 + 通知汇总”时，优先让业务模块返回结果列表，由入口流程统一收集，通知模块再做分组和格式化。
9. 通知结果列表建议使用 `notify_records` 这类直观命名，避免和表格回写 `records` 混淆。

## 排查与修改原则

处理 bug 或异常时，按以下顺序：

1. 先确认当前问题。
2. 再找当前代码中的确定证据。
3. 只修改与问题直接相关的位置。
4. 修改后说明验证方式。
5. 不要把其他项目、旧经验、未确认框架行为套进当前问题。

如果 API、参数、页面行为不确定：

1. 先查 `xbot-api-docs`。
2. 再查当前项目已有代码。
3. 必要时用 `inspect.signature()`、`inspect.getfile()` 查看真实实现。
4. 仍不确定时，明确标注“需运行验证”。

## 市场指令排查规则

当市场指令文档不清楚、参数不明确、可视化能跑但编码版异常时：

1. 先用 `inspect.signature()` 查看包装层参数。
2. 再用 `inspect.getfile()` 定位 `xbot_extensions` 安装目录。
3. 查看 `__init__.py`、`_core.py` 或其他真实实现文件。
4. 以源码里的真实判断逻辑为准。
5. 不要根据界面中文选项猜编码版参数。
6. 若源码判断 `close_type == click`，编码版就应传 `click`，不是 `点击`。

## 影刀项目协作流程

1. 影刀项目由人类先在影刀编辑器中创建，Agent 不负责直接创建 `apps` 目录下的新项目。
2. Agent（Claude Code / Codex / OpenClaw）应直接将真实影刀项目目录作为工作根目录打开，而不是把本知识库仓库当成业务项目目录。
3. Agent 在真实影刀项目目录中开发和修改代码；本仓库只用于查规则、查 API、查示例和查排错方法，不承担业务项目源码目录角色。
4. 影刀项目代码入口通常为 `main(args)`；常见初始化导入包括 `import xbot`、`from . import package`、`from .package import variables as glv`。
5. `args` 中使用的流程参数名必须由人类先在影刀编辑器中配置，Agent 不应自行猜测参数名。
6. 开发完成后，代码不会自动进入影刀编辑器，必须从本仓库根目录调用 `shadowbot_dev_tool.py`，并指定目标影刀项目目录进行同步。
7. 同步前先核对：`main(args)` 入口结构是否正确、`args` 参数名是否已配置、`package.variables` 变量名是否一致、元素库名称和资源文件名称是否一致、目标项目环境是否已安装所需市场指令。
8. 测试由人类在影刀编辑器中完成；Agent 根据测试反馈继续修改并再次同步，不把“已同步”写成“已在编辑器内验证通过”。
9. `shadowbot_dev_tool.py prepare` 只接收 `.py` 文件作为输入；不要把 `package.json` 之类非 Python 文件直接传给 `prepare`。
10. `prepare` 会顺带处理 `package.json` 的 flow 更新和编译；如果只改了 Python 文件，正常只传相关 `.py` 文件即可。
11. `prepare` 或工具落盘后若只看到 `package.json` 换行符变化，通常属于正常现象，不要把纯换行差异误判为业务改动。

最小调用示例：

```powershell
python .\shadowbot_dev_tool.py --project-dir "%LOCALAPPDATA%\ShadowBot\users\<user_id>\apps\<app_id>" prepare main.py
```

## 文档维护约定

1. 本仓库没有统一的构建、启动或测试命令；主要工作流是阅读文档并在影刀环境中验证。
2. 先查文档，再写代码；文档不明确时不要猜。
3. 新增方案应尽量写清业务背景、推荐分工、关键步骤和注意事项。
4. 文档中的 API 引用应链接到 `xbot-api-docs/docs/` 对应文档。
5. 本项目文档主要由历史资料、源码分析、人工整理和 AI 辅助生成，不保证与实际状态始终完全一致。
6. 如果发现文档描述与实际代码行为不符、接口或参数变化、目录结构或配置项变化、文档存在错误遗漏过期内容、或项目新增重要功能但文档尚未覆盖，应记录为待补充事项。
7. 优先完成当前开发任务，不要中断当前工作流去直接改文档；在任务完成后统一汇总问题。
8. 汇报文档差异时，建议包含：发现位置、当前文档描述、实际情况、建议修改内容。
9. 如发现文档与项目实际情况存在差异，主动询问：`发现文档与项目实际情况存在差异，是否需要我同步更新相关文档？`

## LLM Wiki 维护规则

1. `xbot-api-docs/docs/` 是稳定 API 文档层，优先保存经过整理、来源明确、相对稳定的 API 说明。
2. `wiki/` 是 LLM 维护的知识整理层，用于沉淀跨文档总结、常见坑点、错误修正和待验证事项。
3. 新增资料时，不要直接覆盖原始结论；先记录来源、当前判断和验证状态。
4. 发现历史结论错误时，写入 `wiki/error-book.md`，说明错误说法、正确说法、依据和影响范围。
5. 发现暂时无法确认的 API 行为、参数差异、源码差异或业务规则差异时，写入 `wiki/unresolved.md`。
6. 不确定的 API 行为必须标注“需运行验证”，不要包装成确定结论。
7. 确认无误后，再同步更新 `xbot-api-docs/docs/` 中的对应稳定文档。
8. 不要把真实项目路径、账号、token、Cookie 或其他敏感信息写入 Wiki。
9. 能直接作为开发依据的稳定 API 说明，优先写 `xbot-api-docs/docs/`，不要先堆到 `wiki/`。
10. `wiki/` 优先记录跨文档总结、常见坑点、历史纠错和待验证事项，不替代稳定文档主库。
