# CLAUDE.md

本文件是可复制到真实影刀项目根目录 `.claude/CLAUDE.md` 的 Claude Code 协作规则模板。

真实项目同时存在 `AGENTS.md` 时，以根目录 `AGENTS.md` 为主规则入口；本文件用于让 Claude Code 读取同一套核心协作偏好。

## 全局代码生成规则

适用于当前真实影刀项目中的所有代码生成 / 修改任务。

所有代码修改以 **可读、可维护、少工程化、最小改动** 为核心原则。除非用户明确要求复杂封装、重构或长期架构设计，否则优先采用简单、直接、容易审查的实现方式。

## 总体风格

1. 先保证能运行，再考虑优化。
   - 优先给出简单、可执行的方案。
   - 不为“高级”而引入复杂设计。
   - 不处理低概率场景的过度防御。
2. 可读性优先。
   - 变量名直白，逻辑块清晰。
   - 逻辑流程易追踪，减少隐式副作用。
   - 尽量显式数据流，关键状态清楚。
3. 少工程化。
   - 不随意创建类、抽象层或管理器。
   - 只在重复明显或维护成本高时提取函数。
   - 只调用 1 次的函数默认不抽，只有 1 条语句的函数默认不抽。
   - 保持最小可用方案，避免多余封装。
4. 注释简明。
   - 仅标注关键业务逻辑。
   - 函数注释使用 Sphinx / reStructuredText 内联格式。
   - 避免解释 Python 基础语法或冗长文字。
5. 异常处理与返回一致。
   - 异常清楚，函数返回值统一。
   - 避免吞异常或隐藏关键副作用。
6. 兼容与依赖。
   - 不默认兼容旧接口或旧缓存。
   - 尽量使用标准库，第三方库需用户明确允许。

## Python 编码偏好

- 函数优先简单。

```python
def load_cache(file_path):
    ...
```

- 数据操作显式。

```python
cache = get_collect_cache(platform)
cache["items"][key] = item
save_collect_cache(platform)
```

- 全局变量可用但直白。

```python
_COLLECT_CACHE_MAP = {}
```

- 临时变量短但直观。
  - `data`, `item` 可以短。
  - 不要用 `x`, `tmp`, `obj` 这类抽象名。
- 线性流程优先。

```python
if not platform:
    raise ValueError("platform 不能为空")
if not order_no:
    raise ValueError("order_no 不能为空")
```

## 修改代码前流程

1. 确认当前问题。
2. 确认证据来源。
3. 明确应改、不可改部分。
4. 明确最小可行修改。
5. 确认验证方法。
6. 执行修改。

## 禁止行为

- 不要无关重构。
- 不要引入复杂封装。
- 不要擅自更换实现路线。
- 不要引入不必要依赖。
- 不要隐藏副作用。
- 不要吞异常不说明。
- 不要基于旧项目经验做推测。
- 不要改动与当前需求无关的代码。

## 影刀项目规则

1. 写影刀编码版代码前，先查知识库中的 `xbot-api-docs`。
2. 不确定 API 行为时，不要猜，标注“需运行验证”。
3. 市场指令参数不明确时，先排查 `xbot_extensions` 源码。
4. 涉及浏览器、URL、网页业务功能时，除非用户明确要求，不要使用 `requests`、`httpx`、`aiohttp`、`urllib`；只使用 `xbot.web` 及其浏览器对象能力。
5. 能用元素操作就不要优先坐标点击。
6. 中文或长文本输入优先考虑剪贴板输入。
7. 表格大量写入时，优先考虑二维数组批量写入。
8. 不要提交影刀运行包源码、反编译源码、敏感路径、账号信息或 token。
9. 钉钉 AI 表格记录列表默认从 `result.get("data", {}).get("records")` 取，不要直接对顶层结果 `.get("records")`。
10. 钉钉 AI 表格记录默认按 `record["fields"]` 取值；多选字段显示值通常按 `{"name": "...", "id": "..."}` 结构取 `.get("name")`，真正开发前必须实测字段真实结构。

## 代码风格门禁

1. 要不要新函数：只有重复 2 次以上、逻辑已经影响主流程阅读、或用户明确要求复用时才抽函数。如果一个函数只被引用 1 次，默认不要封装；如果函数体只有 1 条语句，默认也不要封装。
2. 要不要新常量：只有多处复用、语义属于公共配置、或提取后明显更清楚时才提常量；单次 URL、字段名、XPath 默认写在使用处。
3. 要不要新类 / 管理器：默认不要。除非现有项目已经用同类结构，或状态复杂到函数无法清楚表达。
4. 要不要配置层：默认不要。当前任务只用一次的参数直接写在调用处或从既有 `args` / `package.variables` 读取。
5. 要不要兼容多种输入：调用方已有约定时，直接按约定结构使用，不反复写防御性归一化。
6. 要不要捕获异常：只有当前业务需要补充上下文、通知、截图或恢复流程时才捕获；捕获后必须保留关键错误信息。
7. 要不要重试 / 多候选兜底：默认不要。只有页面或接口已确认存在这种不稳定性时才加。
8. 要不要顺手改名 / 格式化 / 重排代码：默认不要，只改当前需求直接相关的行。

## 通知与结果

1. 涉及“业务处理 + 通知汇总”时，优先让业务模块返回结果列表，由入口流程统一收集，通知模块再做分组和格式化。
2. 通知结果列表建议使用 `notify_records` 这类直观命名，避免和表格回写 `records` 混淆。
3. 生成代码后必须回看一遍：如果删掉某个抽象、常量或兜底分支后代码仍然清楚，就删掉它。
4. 阅读性优先于工程化；如果内联后主流程更直观，就不要为了“结构更漂亮”再包一层函数。

## 收尾要求

1. 删除本次改动造成的不再使用的 import、变量或函数。
2. 说明验证方式；不能在影刀编辑器内验证时，不要写成“已验证通过”。
3. 真实影刀项目中修改 `.py` 后，必须运行知识库根目录的 `shadowbot_dev_tool.py prepare` 同步收尾。
4. `shadowbot_dev_tool.py prepare` 只接收 `.py` 文件作为输入；不要把 `package.json` 之类非 Python 文件直接传给 `prepare`。
5. `prepare` 会顺带处理 `package.json` 的 flow 更新和编译，但不会自动备份项目；如需备份，显式调用 `shadowbot_dev_tool.py backup ...`。

最小调用示例：

```powershell
python C:\Users\Administrator\Desktop\影刀xAI开发指南\shadowbot_dev_tool.py --project-dir "%LOCALAPPDATA%\ShadowBot\users\<user_id>\apps\<app_id>" prepare main.py
```

## 核心原则

`最小改动 | 结构简单 | 流程显式 | 命名直白 | 少工程化 | 易审查 | 易维护`
