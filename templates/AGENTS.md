# AGENTS.md

本文件是可复制到真实影刀项目根目录的通用 Agent 规则模板，适用于 OpenClaw、Codex、Claude Code 等 Agent。

本模板不参与 `xbot-AI-docs` 知识库检索；知识库自身规则以根目录 `AGENTS.md` 为准。

## 全局代码生成规则

适用于当前真实影刀项目中的所有代码生成 / 修改任务。

所有代码修改以 **可读、可维护、少工程化、最小改动** 为核心原则。除非用户明确要求复杂封装、重构或长期架构设计，否则优先采用简单、直接、容易审查的实现方式。

## 总体风格

1. 先保证能运行，再考虑优化。
   - 优先给出简单、可执行的方案。
   - 不为“高级”而引入复杂设计。
   - 不处理低概率场景的过度防御。
2. 可读性优先。
   - 变量名直白，逻辑块清晰。
   - 逻辑流程易追踪，减少隐式副作用。
   - 尽量显式数据流，关键状态清楚。
3. 少工程化。
   - 不随意创建类、抽象层或管理器。
   - 只在重复明显或维护成本高时提取函数。
   - 保持最小可用方案，避免多余封装。
4. 注释简明。
   - 仅标注关键业务逻辑。
   - 函数注释使用 Sphinx / reStructuredText 内联格式。
   - 避免解释 Python 基础语法或冗长文字。
5. 异常处理与返回一致。
   - 异常清楚，函数返回值统一。
   - 避免吞异常或隐藏关键副作用。
6. 兼容与依赖。
   - 不默认兼容旧接口或旧缓存。
   - 尽量使用标准库，第三方库需用户明确允许。

## Python 编码偏好

- 函数优先简单。

```python
def load_cache(file_path):
    ...
```

- 数据操作显式。

```python
cache = get_collect_cache(platform)
cache["items"][key] = item
save_collect_cache(platform)
```

- 全局变量可用但直白。

```python
_COLLECT_CACHE_MAP = {}
```

- 临时变量短但直观。
  - `data`, `item` 可以短。
  - 不要用 `x`, `tmp`, `obj` 这类抽象名。
- 线性流程优先。

```python
if not platform:
    raise ValueError("platform 不能为空")
if not order_no:
    raise ValueError("order_no 不能为空")
```

- 清理过期缓存时，注释只说明业务目的。

```python
# 只保留最近 CACHE_KEEP_DAYS 天的记录
```

## 修改代码前流程

2. 确认证据来源。
3. 明确应改、不可改部分。
4. 明确最小可行修改。
5. 确认验证方法。
6. 执行修改。

## 禁止行为

- 不要无关重构。
- 不要引入复杂封装。
- 不要擅自更换实现路线。
- 不要引入不必要依赖。
- 不要隐藏副作用。
- 不要吞异常不说明。
- 不要基于旧项目经验做推测。
- 不要改动与当前需求无关的代码。

## 使用顺序

1. 先确认当前工作目录是真实影刀项目根目录，不是知识库或模板库目录。
2. 写影刀编码版代码前，先查知识库中的 `xbot-api-docs`。
3. 浏览器操作查知识库中的 `xbot-api-docs/docs/browser.md`。
4. 表格操作查知识库中的 `xbot-api-docs/docs/excel.md`。
5. 市场指令参数不明确时，查知识库中的 `xbot-api-docs/docs/debug/market-extension-source.md`。

## 使用原则

1. 不确定 API 行为时，不要猜，标注“需运行验证”。
2. 市场指令参数不明确时，先按知识库中的 `xbot-api-docs/docs/debug/market-extension-source.md` 的方法排查源码。
3. 价格采集优先使用 XPath、CSS、页面接口或明确规则取数，不要让 Agent 自由猜页面价格。
4. 涉及浏览器、URL、网页业务功能时，除非用户明确要求，不要使用 `requests`、`httpx`、`aiohttp`、`urllib` 或其他网络相关 Python 库；只使用 `xbot.web` 及其浏览器对象能力处理页面、请求、Cookie、网络监听和下载。
5. 代码优先保证可读性，变量名、流程和判断逻辑要直观。
6. 不要过度工程化，不要为了简单任务引入复杂封装、抽象层、配置系统或通用框架。
7. 只基于当前业务、当前代码、当前页面和当前需求修改。
8. 不要优化未知问题，不要提前处理未出现、未验证、低概率的场景。
9. 不要做无关重构，不要改动与当前任务无关的代码、格式、命名或结构。
10. 修改代码时优先做最小可行改动，确保每一处改动都能对应当前需求。
11. 如果发现潜在问题，可以指出，但不要擅自扩展修改范围。
12. 页面字段、价格口径、按钮含义必须以当前业务规则为准，不要自行推断。
13. 可视化界面能跑不代表编码版参数一致，编码版参数必须以文档或源码为准。
14. 能用元素操作就不要优先坐标点击。
15. 中文或长文本输入优先考虑剪贴板输入。
16. 表格大量写入时，优先考虑二维数组批量写入。
17. 不要提交影刀运行包源码、反编译源码、敏感路径、账号信息或 token。
18. 不要编造不存在的功能、命令、目录、文件或 API。
19. 钉钉 AI 表格记录列表默认从 `result.get("data", {}).get("records")` 取，不要直接对顶层结果 `.get("records")`。
20. 钉钉 AI 表格记录默认按 `record["fields"]` 取值；多选字段先按 `{"name": "...", "id": "..."}` 结构理解显示值，真正开发前必须实测字段真实结构。

## 代码风格要求

1. 代码要简单、直接、容易阅读。
2. 优先顺序式流程，不要为了少量逻辑拆很多函数。
3. 只有当重复明显、逻辑复杂或当前任务明确需要时，才抽函数。
4. 单次使用的常量、URL、固定字符串、小函数、小表达式默认直接写在使用处，不要因为“看起来像常量”就单独提取。
5. 调用方已明确约定入参类型或记录结构时，被调函数直接按约定使用；不要在业务逻辑中重复写 `isinstance`、`dict(...)`、空结构回退这类防御性归一化。
6. 如果输入形态确实不确定，只允许在边界统一归一化一次；进入业务逻辑后，不要继续重复判断和兜底。
7. 不要默认添加复杂异常体系、重试框架、日志框架或配置层。
8. 异常处理只覆盖当前业务中明确需要处理的失败情况。
9. 注释只解释业务关键点、特殊页面逻辑或不直观的 API 行为，不要写无意义注释。
10. 保持现有项目风格，不要无关格式化整个文件。
11. 涉及“业务处理 + 通知汇总”时，优先让业务模块返回结果列表，由入口流程统一收集，通知模块再做分组和格式化。
12. 通知结果列表建议使用 `notify_records` 这类直观命名，避免和表格回写 `records` 混淆。
13. 生成代码后必须回看一遍：如果删掉某个抽象、常量或兜底分支后代码仍然清楚，就删掉它。

## 代码风格门禁

写代码前先做以下判断；只要答案是否定，就不要写对应结构：

1. 要不要新函数：只有重复 2 次以上、逻辑已经影响主流程阅读、或用户明确要求复用时才抽函数。
2. 要不要新常量：只有多处复用、语义属于公共配置、或提取后明显更清楚时才提常量；单次 URL、字段名、XPath 默认写在使用处。
3. 要不要新类 / 管理器：默认不要。除非现有项目已经用同类结构，或状态复杂到函数无法清楚表达。
4. 要不要配置层：默认不要。当前任务只用一次的参数直接写在调用处或从既有 `args` / `package.variables` 读取。
5. 要不要兼容多种输入：调用方已有约定时，直接按约定结构使用，不反复写防御性归一化。
6. 要不要捕获异常：只有当前业务需要补充上下文、通知、截图或恢复流程时才捕获；捕获后必须保留关键错误信息。
7. 要不要重试 / 多候选兜底：默认不要。只有页面或接口已确认存在这种不稳定性时才加。
8. 要不要顺手改名 / 格式化 / 重排代码：默认不要，只改当前需求直接相关的行。

## 排查与修改原则

处理 bug 或异常时，按以下顺序：

1. 先确认当前问题。
2. 再找当前代码中的确定证据。
3. 只修改与问题直接相关的位置。
4. 修改后说明验证方式。
5. 不要把其他项目、旧经验、未确认框架行为套进当前问题。

如果 API、参数、页面行为不确定：

1. 先查知识库中的 `xbot-api-docs`。
2. 再查当前项目已有代码。
3. 必要时用 `inspect.signature()`、`inspect.getfile()` 查看真实实现。
4. 仍不确定时，明确标注“需运行验证”。

## 市场指令排查规则

当市场指令文档不清楚、参数不明确、可视化能跑但编码版异常时：

1. 先用 `inspect.signature()` 查看包装层参数。
2. 再用 `inspect.getfile()` 定位 `xbot_extensions` 安装目录。
3. 查看 `__init__.py`、`_core.py` 或其他真实实现文件。
4. 以源码里的真实判断逻辑为准。
5. 不要根据界面中文选项猜编码版参数。
6. 若源码判断 `close_type == click`，编码版就应传 `click`，不是 `点击`。

## 影刀项目协作流程

1. 影刀项目由人类先在影刀编辑器中创建，Agent 不负责直接创建 `apps` 目录下的新项目。
2. Agent 应直接将真实影刀项目目录作为工作根目录打开，而不是把知识库仓库当成业务项目目录。
3. 影刀项目代码入口通常为 `main(args)`；常见初始化导入包括 `import xbot`、`from . import package`、`from .package import variables as glv`。
4. `args` 中使用的流程参数名必须由人类先在影刀编辑器中配置，Agent 不应自行猜测参数名。
5. 开发完成后，代码不会自动进入影刀编辑器，真实影刀项目中修改 `.py` 后必须运行知识库根目录的 `shadowbot_dev_tool.py prepare` 同步收尾。
6. 同步前先核对：`main(args)` 入口结构、`args` 参数名、`package.variables` 变量名、元素库名称、资源文件名称、目标项目环境是否已安装所需市场指令。
7. 测试由人类在影刀编辑器中完成；Agent 根据测试反馈继续修改，不把“已同步”写成“已在编辑器内验证通过”。
8. `shadowbot_dev_tool.py prepare` 只接收 `.py` 文件作为输入；不要把 `package.json` 之类非 Python 文件直接传给 `prepare`。
9. `prepare` 会顺带处理 `package.json` 的 flow 更新和编译，但不会自动备份项目；如需备份，显式调用 `shadowbot_dev_tool.py backup ...`。

最小调用示例：

```powershell
python C:\Users\Administrator\Desktop\影刀xAI开发指南\shadowbot_dev_tool.py --project-dir "%LOCALAPPDATA%\ShadowBot\users\<user_id>\apps\<app_id>" prepare main.py
```
